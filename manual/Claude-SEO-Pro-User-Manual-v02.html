1 · Welcome — what you're holding

Claude SEO Pro turns Claude Code into a complete SEO audit department: 30 skills, 19 specialist agents, live data from DataForSEO and Google, persistent per-client memory, and client-grade PDF reports — run by one person, from one command line, with every number in every report traceable to evidence.

The one story that explains every design decision

An early audit shipped findings that turned out to be fabricated. Subagents couldn't fetch the website, silently substituted "typical patterns for sites of this type," and the orchestrator stripped their warning caveats — promoting guesses to "Critical Issues." The client caught it.

Everything distinctive about this system is a countermeasure to that incident:

  • Evidence Integrity Protocol — every finding cites a captured file and line, or a named API response field. No source → it doesn't ship.
  • "Data pending", never guesses — a blocked API produces an honest pending section with the exact error code and the unlock step.
  • A machine-enforced report contract — a linter rejects reports that miss sections or compress into summaries.
  • Receipts — every report's appendix lists every API call it made, with per-call status.
What this means for you
When a client asks "how do you know?", the answer is in the report. That is the product.

4 · Core concepts (10 minutes that make everything else obvious)

4.1 Architecture: skills, agents, engines

Three kinds of moving parts:

PartWhat it isExamples
Skills (30)Instruction packs that teach Claude a workflow. You invoke them as /seo … commands.seo-audit, seo-schema, Pro seo-setup, seo-knowledge
Agents (19)Specialist sub-workers a skill spawns in parallel — each analyzes one dimension (technical, content, schema…) and reports back with evidence.seo-technical, seo-content, Pro seo-learn
Engines (4)Pro Python packages that do deterministic work: onboarding/credentials, client knowledge, model routing, the Slack connector.onboarding/, knowledge/, routing/, connector/

Where things live on your machine after install:

~/.claude/skills/          ← the 30 skills (seo, seo-audit, seo-setup, …)
~/.claude/skills/seo/      ← shared scripts + the 4 Pro engines
~/.claude/agents/          ← the 19 agent definitions
~/.config/claude-seo/      ← YOUR data: API keys (owner-only), client knowledge,
                              cache, branding.json, install-manifest.json
The golden separation
The repo (claude-seo-pro folder) is the product. ~/.claude is the installed copy. ~/.config/claude-seo is your private data — keys and client knowledge. The repo never contains your data; updates never touch your data.

4.2 The Evidence Integrity Protocol

Eight rules wired into the audit skill itself. Worth knowing them, because they explain "odd" behaviors that are actually the system protecting you:

The 8 rules, and what each one looks like in practice
#RuleWhat you'll observe
1Pre-fetch all pages first; specialists read local files onlyThe audit starts by downloading dozens of pages to disk before any analysis
2A failed fetch is reported, never substituted"Could not fetch /pricing (403)" instead of invented findings about the pricing page
3Every finding cites evidence (file:line or API field)Each finding card has a small "Evidence:" line
4Specialist caveats are never strippedYou may see [VERIFY] markers survive into drafts — that's intentional
5No score without backing dataBacklinks shows PENDING in the scorecard rather than a made-up number
6CWV uses real-user field data; lab labeled as lab"Lab LCP 18s (throttling artifact — field is 2.4s)"
7Schema is read from raw HTMLThe fetcher saves raw HTML because rendered fetches strip JSON-LD
8What was NOT covered is reported"Crawl sampled 50 of 181 sitemap URLs" in the appendix

4.3 The 4-phase audit pipeline

A full /seo audit is not one step — it's four phases, in order:

PhaseNameWhat happensOutput
0Business IntelligenceRecalls everything already known about this client from the knowledge store, then infers business model, country, target markets, ideal customers, seed keyword themes — each with confidence + evidencebusiness-profile.json
1Evidence auditPre-fetches all money pages + sitemaps + robots; spawns parallel specialists (cheap models for extraction, stronger for reasoning); adversarially verifies Critical/High findingsper-dimension findings, all cited
2Keyword researchLive DataForSEO across the origin + detected markets: volumes, CPC, difficulty, intent, competitor gap; tiered into quick-wins / rescues / gapskeyword tables + opportunity tiers
3Local / GBPOnly when the client is a local business: live Business Profile data, NAP consistency, reviews (skips itself with a note for national B2B firms)GBP table or skip note
→Report + write-backAssembles the 14-section report, lints it, generates the PDF, then the learning agent stores durable facts for next timereport .md/.html/.pdf + updated memory

4.4 Client memory (why audit #2 is better than audit #1)

The system keeps a persistent, per-client knowledge store on disk — independent of any chat session or Claude model:

  • Profile — what the business is (model, markets, ICPs, seed themes)
  • Facts — durable, evidence-tagged learnings ("authors are CSS-hidden sitewide — evidence: 4 files cited"), each with confidence + source. Unsourced claims are force-downgraded to low confidence; transient metrics and anything secret-shaped are rejected.
  • History — the audit-score timeline (48 → 57 → …)
  • Cache — expensive API responses with freshness windows + provenance, so repeat audits don't re-pay for unchanged data and cached numbers stay citable

Every audit reads this first (starts informed) and writes back after (gets smarter). See §11 to drive it manually.

5.2 Verify the install + the drift guard New

From the repo folder, anytime:

python tools/check_install.py

It hashes every Pro-owned file in the repo against the installed copy and prints one of:

  • [INSTALL FRESH] installed skills match the repo. — you're current.
  • [INSTALL STALE] re-run install.ps1 / install.sh — with a list of DRIFTED/MISSING files.
Why this exists — a true story
A real audit once ran against stale installed skills because the installer wasn't re-run after the product gained new features. The audit still "worked" — on the old playbook. The drift guard makes that condition loud instead of silent. Habit: run it after every git pull.

/seo-setup verify also surfaces this check automatically when run from a repo checkout.

5.3 Updating to a new version

cd claude-seo-pro
git pull
python tools/check_install.py        # will report STALE
powershell -ExecutionPolicy Bypass -File install.ps1 -NoOnboard   # refresh
python tools/check_install.py        # should now report FRESH

Your keys, client knowledge, cache and branding are in ~/.config/claude-seo/ — untouched by updates.

Uninstall

Delete the skill/agent folders the installer created under ~/.claude/skills/ (all seo* directories) and ~/.claude/agents/seo-*.md. To remove your data too (keys, client memory): delete ~/.config/claude-seo/ — this is irreversible; the client knowledge store lives there.

6 · API onboarding — all 7 providers, step by step

Two equivalent ways to run onboarding; both end up in the same secure store:

/seo-setup                                              ← in Claude Code (guided, recommended)
python ~/.claude/skills/seo/onboarding/setup_wizard.py  ← directly in your terminal
Security model in one line
Keys are typed into a masked terminal prompt (never into chat), validated live against the provider before storage, then written to ~/.config/claude-seo/*.json with owner-only permissions. Nothing is sent anywhere except the provider's own API. /seo-setup rotate <provider> replaces a key later.

Status check anytime: /seo-setup verify or python …/setup_wizard.py --check. Configure a single provider: --provider dataforseo etc.

6.1 DataForSEO — the off-site engine (most valuable single key)

Unlocks: rankings, full keyword research (volume/CPC/difficulty/intent), competitor analysis & gap, business listings, AI-mention checks. Without it the audit is on-site only.

1
Create an account at app.dataforseo.com/register. Top up a small balance ($10–50 lasts a long time — a full audit's data is typically $0.10–$3).
2
Your credentials are the API login (email) + API password shown on the dashboard — not your site login password.
3
Run the wizard; it validates with a $0 auth call and a ~$0.01 data probe.
THE classic trap: IP whitelist (error 40207)
DataForSEO accounts often have an IP-whitelist enabled. Your key can be valid while every data call is blocked until your machine's public IP is added at app.dataforseo.com/api-access. The validator detects this distinctly and tells you. On home/mobile connections your IP rotates — it will break again. Either disable the whitelist, or expect to re-add the new IP (find it: curl https://api.ipify.org). Full play-by-play: §17.
What the audit can and can't do at each DataForSEO subscription level
You haveWorksShows as "Data pending"
Base account + balanceRankings, keyword research, competitors, listings, AI mentionsBacklinks (needs the separate Backlinks subscription — error 40204)
+ Backlinks subscriptionEverything incl. referring domains / domain rating—

6.2 Google API key — real Core Web Vitals (free)

Unlocks: PageSpeed Insights + CrUX field data — what real Chrome users experience (the ranking signal), not just lab simulations. Free with normal quotas.

1
Go to console.cloud.google.com/apis/credentials → create/select a project → Create credentials → API key.
2
Enable the PageSpeed Insights API for the project (APIs & Services → Library → search "PageSpeed").
3
Run the wizard — it validates the key without burning a full Lighthouse run.

6.3 Google Search Console + GA4 (OAuth) — skippable, attach later

Unlocks: indexation status, search performance (clicks/impressions/CTR/position), GA4 organic traffic trends. Needs owner access to the client's properties, so the wizard supports [N]ow / [L]ater / [S]kip — "Later" marks it pending and audits degrade gracefully ("Data pending — Google OAuth not connected").

Full OAuth setup when you're ready (10 minutes, one time)
1
In the same Google Cloud project: enable Search Console API and Google Analytics Data API.
2
OAuth consent screen → External → add yourself as a test user.
3
Credentials → Create → OAuth client ID → Desktop app → download client_secret.json.
4
Run the wizard with the file path; a browser opens for consent; the token is stored owner-only. The client_secret itself is read from your file at runtime, never copied into the token.

6.4 Google Business Profile (OAuth) — for local-business clients

Unlocks: first-party local insights (impressions, calls, direction requests). Needs the profile owner's consent. If skipped, local audits automatically fall back to DataForSEO's public listing data — labeled accordingly. Same OAuth pattern as §6.3; additionally requires GBP API access approval from Google (the wizard explains).

6.5 Firecrawl — large/JS-heavy site crawling

Unlocks: full-site crawling with JavaScript rendering — for SPAs and very large sites where the built-in fetcher isn't enough. Key from firecrawl.dev/app/api-keys; the validator checks your credit balance.

6.6 Exa — neural search for competitor/entity research

Unlocks: semantic web search used in competitor discovery and entity research. Key from dashboard.exa.ai/api-keys; validated with a single 1-result probe.

6.7 Slack — the chat connector

Unlocks: running audits from Slack (/seo audit client.com in a channel). Needs a Slack app's signing secret + bot token; the validator calls auth.test. Full deployment walkthrough in §13 — configure the provider here first, deploy the bridge there.

Minimum viable setup
Just DataForSEO + Google API key gives you the full 4-phase audit with live rankings, keyword research and real CWV. Everything else adds depth and can be attached later without re-doing anything.

7 · Your first full audit — a complete walkthrough

Type this in Claude Code and press Enter:

/seo audit https://yourclient.com

Here is what happens, phase by phase, with what you should expect to see — so you can tell healthy progress from a problem:

Phase 0 — Business intelligence (1–2 min)

Claude first recalls the client's knowledge store ("What we know about X… 12 learned facts… last score 57" — or "no prior knowledge" on a first run), then infers the business profile from the homepage and confirms anything uncertain with you. You may be asked 1–2 questions ("Is KSA a real target market or just a translated page?") — answering well here makes Phase 2's keyword research dramatically better targeted.

Phase 1 — Evidence capture + parallel specialists (5–10 min)

You'll see pages being fetched to disk (dozens of URLs, each with an HTTP status), then several specialist agents launching in parallel. Expect names like seo-technical, seo-content, seo-schema, seo-geo with running token counts. Cheap models handle extraction; stronger ones do reasoning — this is the model router (§12) saving you money automatically.

Healthy signs
"50 fetches, 50 OK" · agents citing file paths · "adversarially verifying finding…"
Warning signs
fetch failures piling up (site blocking — the audit will say so honestly) · an agent reporting it cannot read its evidence (see §17).

Phase 2 — Live keyword research (2–5 min, ~$0.05–$1)

DataForSEO calls fan out: your rankings, position distribution, competitor set, keyword demand, the gap vs a real competitor. Every response is saved to disk — those files are the citations behind the report's tables.

Phase 3 — Local/GBP (skips itself for non-local businesses)

National B2B firm? You'll see a one-line skip note (that's contract-correct, not an omission). Local business? Live GBP data: rating, reviews, categories, photos, NAP comparison.

Assembly → linter → PDF → memory write-back (2–4 min)

The report is assembled to the 14-section contract, the linter checks it (§8.2), the PDF is generated, and the learning agent proposes durable facts to remember (preview-then-ingest). The run ends by stating where every deliverable is.

DeliverableWhat it's for
FULL-AUDIT-REPORT.mdWorking markdown — grep-able, diff-able
ACTION-PLAN.mdPrioritized actions with effort estimates
<client>-seo-audit-<YYYY-MM>.pdfThe client deliverable — white-labeled, contract-complete
Evidence folder + saved API responsesThe receipts — keep them with the engagement
Knowledge-store updateAudit #2 starts informed
Cost anatomy of a real audit (receipts from production)

A real full live-data audit (Dubai client, 5 locales): 28 DataForSEO calls, $0.61 total. A US B2B audit's data: ~$0.10. Claude token costs depend on site size; the router keeps extraction on the cheapest tier. Preview keyword-research cost before spending: python ~/.claude/skills/seo/scripts/keyword_research.py --plan --seeds "…" --markets "US:en".

8 · Reading & delivering the report

8.1 The 14 sections, and how to read each one

#SectionHow to read it / what to check
1CoverClient, scope, version, date. White-labeled with your brand (§8.4).
2Data Integrity & MethodologyRead this first with a skeptical client. How evidence was gathered; every pending data source with its exact error + unlock step.
3Executive SummaryThe story in one page: 4 KPI cards, the score ring, the full category scorecard (PENDING rows greyed — honesty, not failure), the 5 highest-leverage moves.
4Priority IssuesFinding cards: severity → explanation → verbatim evidence block → fix → source. If a finding has no evidence line, it shouldn't be there — that's the contract.
5What's Genuinely StrongMandatory. A findings-only report reads as a hit job and loses the client.
6Organic Search VisibilityPosition-distribution buckets (each with a "read" column) + the ≥10-row quick-wins table. Watch for the honesty pattern: quick-wins labeled "brand/noise" when that's the truth.
7Keyword ResearchSeeds-by-market volumes/CPC, the competitor-gap table, opportunity tiers. Thin secondary markets are said to be thin.
8Local SEO & GBPField-by-field GBP table + cross-source NAP check — or an explicit one-line skip note for non-local clients.
9Competitive LandscapeWho shares your keyword footprint, classified (direct competitor vs directory/platform — you don't out-rank Wikipedia, you learn from it).
10On-Page DetailOne row per audited money page — all of them. Title/desc lengths, H1s, per-page issues, cannibalization patterns, and what's confirmed clean.
11Performance (field vs lab)Two separate tables. Field = the ranking signal; lab = diagnostic only. Lab artifacts are labeled (an 18s lab LCP next to a 2.4s field LCP is throttling, not reality).
12Per-category detailContent/E-E-A-T, Schema, AI-readiness each get full depth — this is where the specialists' work lands.
13Action PlanQuick-wins + strategic tables (effort + data-backed why) + sequencing + the "data to unlock" list.
14AppendixThe receipts: every API call with status, integrity controls, evidence corpus, corrections vs any prior report.

8.2 The contract linter New

Before any report ships, lint it (the audit does this automatically; you can run it yourself on any report):

python tools/lint_report.py path\to\report.html          # human verdict
python tools/lint_report.py path\to\report.html --json   # machine-readable
VerdictMeaningYour move
PASSAll 14 sections present, no leftover placeholders, depth floors metDeliver
WARNStructure OK; depth floors or pending-cause notes flaggedReview the warnings, then deliver
FAILMissing sections / unfilled placeholders / summary-only compressionDo not deliver. Fix, re-lint

11 · Client knowledge & learning — the system that gets smarter

Conversational use is §9 (/seo-knowledge, /seo-learn). Under the hood, per client (slug of the domain):

~/.config/claude-seo/clients/yourclient-com/
  profile.json    ← business understanding (model, markets, ICPs, seeds)
  facts.jsonl     ← evidence-tagged durable facts (append-only, dedup, supersede)
  history.jsonl   ← audit-score timeline
~/.config/claude-seo/cache/<provider>/   ← cached API responses + provenance

What makes a fact "durable" (the guards)

CandidateOutcomeWhy
"Targets KSA as a secondary market" + evidence✅ stored (high)Durable + sourced
"Targets enterprise clients" (no evidence)⚠ stored at lowUnsourced → forced low confidence
"Performance score is 30 right now"❌ rejectedTransient metric → belongs in history, not facts
"API key is AIza…"❌ rejectedSecret-shaped content never enters memory

When the truth changes, facts are superseded, not duplicated — memory gets more accurate, not just bigger. The cache refuses stale reads (expired = miss = re-fetch) and keeps provenance so even cached numbers stay citable.

Power use: the CLI underneath 
python ~/.claude/skills/seo/knowledge/store.py recall yourclient.com
python ~/.claude/skills/seo/knowledge/store.py add-history yourclient.com --score 61 --note "post-CLS-fix re-audit"
python ~/.claude/skills/seo/knowledge/learn.py preview --domain yourclient.com --file candidates.json
python ~/.claude/skills/seo/knowledge/cache.py stats
Back this up
~/.config/claude-seo/clients/ is your accumulated client intelligence. Include it in your backups (see §16). It is deliberately NOT in the repo.

12 · Model routing & cost control

Each audit task runs on the cheapest Claude model that does it well, decided at dispatch time. The main conversation stays on your session's model (switching it would invalidate the prompt cache — slower and pricier).

TierModelAudit work
extractionHaiku 4.5Titles/metas/H1s/status/schema-presence from pre-fetched files
verificationSonnet 4.6Adversarial double-check of Critical/High findings
reasoningSonnet 4.6E-E-A-T judgment, intent analysis, fact extraction
synthesisOpus 4.8Cross-dimension prioritization, the report itself
orchestrationOpus 4.8 (pinned)The main loop

Real example: a 60k-token extraction task costs $0.11 on Haiku vs $0.55 on Opus — multiplied across every specialist, every audit. Inspect and override:

/seo-models                                                   # show policy
python ~/.claude/skills/seo/routing/model_router.py route --agent seo-technical
python ~/.claude/skills/seo/routing/model_router.py estimate --tier extraction --in 60000 --out 10000
python ~/.claude/skills/seo/routing/model_router.py set --force-model opus    # flagship-quality pass
python ~/.claude/skills/seo/routing/model_router.py reset
Why cheap models are safe here
Extraction agents read pre-fetched files — there's nothing to hallucinate about; they're transcribing. Judgment work is routed up-tier. One observed limit: very large minified pages (300KB+) can exhaust a small model's turn budget — if a specialist stalls, that's the first suspect (§17).

13 · Slack connector — audits from chat

For the manager who doesn't live in a terminal: type /seo audit client.com in Slack; the bridge verifies, authorizes, acknowledges in under 3 seconds, runs a headless audit, and posts the result back.

Security model (read before deploying)

  • Signature verification — every request HMAC-checked against your signing secret with a 5-minute replay window; forged/stale requests get 401.
  • Deny-by-default authorization — nobody can trigger audits until you allow-list their user/channel ID. Audits cost money; this is intentional.
  • Secrets stay host-side — the bot token/signing secret live in ~/.config/claude-seo/slack.json on the machine running the bridge, never in Slack.
  • Fixed command surface — only commands you enable parse at all; URL targets are validated.
Deployment walkthrough (one-time, ~30 min)
1
Create a Slack app (api.slack.com/apps) → add a slash command /seo pointing at your bridge URL → install to workspace. Note the signing secret and bot token.
2
Onboard the slack provider (§6.7) so the secrets are stored.
3
Configure operating policy in ~/.config/claude-seo/connector.json: enabled_commands (e.g. ["audit","page"]), allowed_users (Slack user IDs), model, timeout.
4
Run the bridge on a host that has Claude Code installed and is reachable by Slack (or via a tunnel): 
python ~/.claude/skills/seo/connector/slack_bridge.py --port 8743
python ~/.claude/skills/seo/connector/slack_bridge.py --selftest   # offline sanity first
5
In Slack: /seo audit https://client.com → expect the "running…" ack, then the result post. Unauthorized users get a polite denial (by design).

Full deploy doc: docs/CONNECTOR.md. WhatsApp: planned as an adapter on the same runner.

Honest status
The connector's security core (signatures, authorization, parsing, runner) is unit-tested in CI. The live Slack round-trip is your deployment step — it needs your Slack app and a reachable host.

14 · Drift monitoring — "did anything break?"

Git for your client's on-page SEO. Capture a baseline after every audit/fix cycle; compare any time something ships:

/seo drift baseline https://client.com     # snapshot SEO-critical elements (SQLite)
/seo drift compare  https://client.com     # diff current site vs baseline, severity-tagged
/seo drift history  https://client.com     # the change timeline

The workflow that earns retainers: baseline after delivery → client's dev team ships things → compare before your monthly check-in → walk in saying "your release on the 14th dropped canonical tags on 12 pages" with the diff to prove it. Pair with the knowledge store's score history for the long-run trend conversation.

15 · Staying current — sync, CI, versions

Upstream sync (you benefit from the open-source project forever)

python tools/sync_upstream.py --dry-run      # preview upstream changes (also the provenance check)
python tools/sync_upstream.py --tag vX.Y.Z   # vendor a new upstream release

Pro-owned files are protected by a skip-list; the one in-place patch re-applies as an idempotent overlay that fails loudly if upstream moves an anchor. A weekly GitHub Action does this automatically and opens a reviewable PR — never auto-merges. A clean dry-run reports exactly one changed file (the managed overlay): that's your proof the fork is still upstream + overlay + additions.

CI New

Every push/PR runs: the 68-assertion adversarial component suite · repo integrity (compile-all, JSON validity, secret scan, no client data) · overlay check · report-linter self-test. All stdlib, offline, no keys needed. Check the repo's Actions tab — green before you pull, green before you deploy to a client seat.

Versioning

VERSION.md is the human ledger (what's upstream vs Pro, version history); system-version.json is the machine manifest; the installed stamp is in ~/.config/claude-seo/install-manifest.json. The roadmap — including the refused-features list (no hosted dashboard, no own crawler, no multi-tenant SaaS: they'd destroy the local-first differentiation) — is docs/ROADMAP-7-to-9.md.

19 · FAQ

Do I need to know how to code?

No. You type /seo commands in Claude Code and occasionally paste a command into a terminal (this manual gives you every one verbatim). The system was built for exactly this profile.

Does it work with zero API keys?

Yes — the on-site audit (technical, content, schema, on-page, AI-readiness) runs from captured HTML alone. Keys add the off-site picture; each missing source becomes an honest "Data pending" section, never a guess.

What does an audit cost?

API data: typically $0.10–$3 (real receipt: 28 calls, $0.61, five locales). Claude tokens: depends on site size; the model router keeps the bulk on the cheapest tier. Preview keyword costs before spending with --plan.

Why does the report say "Data pending" instead of just showing a number?

Because the alternative is making the number up. Pending sections carry the exact error and the unlock step — fix the access, re-run, the section fills in. This is the product's core promise.

Can I rebrand reports for my agency?

Yes — §8.4. One command sets preparer/colors/logo/footer; every later report uses it. No per-report work, no template editing.

How do upstream improvements reach me?

A weekly Action vendors the latest upstream release, protects all Pro files, re-applies the managed patch, and opens a PR for your review. Manually: python tools/sync_upstream.py --tag vX.Y.Z.

Multiple clients — do they see each other's data?

No. Knowledge is stored per client (per-domain folders). Reports are generated per client. Nothing is multi-tenant because nothing is hosted — it's all on your machine.

Is my client's data sent anywhere?

Page content goes to Anthropic (that's Claude analyzing it) and queries go to the data providers you enabled (DataForSEO, Google). Nothing else. No analytics, no telemetry, no vendor cloud holding your audits.

What's deliberately NOT in this product?

A hosted dashboard, multi-tenant SaaS, our own crawler, SLAs — see the refused-features list in docs/ROADMAP-7-to-9.md. They'd destroy the local-first differentiation that makes this trustworthy.
